# kre8 -- Core Schema Definitions

This document defines the **initial schema contracts** used inside the
kre8 Thinking Platform Layer (TPL).

These schemas represent the **core data structures** exchanged between
internal components.

Current schemas:

1.  StructuredIntent
2.  DesignPlan

------------------------------------------------------------------------

# 1. StructuredIntent Schema

StructuredIntent represents **normalized user intent** extracted from
natural language input.

It is produced by the **intent parsing pipeline**.

Pipeline:

User NLP\
↓\
Heuristic Signal Extraction\
↓\
LLM Intent Extraction\
↓\
Schema Validation\
↓\
StructuredIntent

Purpose:

-   Normalize user requests
-   Provide structured input for the design engine
-   Remove ambiguity from raw natural language

Example Structure:

    StructuredIntent
       goal
       scope
       signals
       constraints_hint
       questions
       confidence

Field Definitions:

goal\
Short normalized statement of the user's objective.

scope\
Optional scope information such as environment, team, or platform.

signals\
Signals extracted from heuristic parsing.

Examples:

-   regions
-   cloud provider hints
-   kubernetes / cluster indicators
-   service mesh keywords

constraints_hint\
User‑mentioned constraints that may influence design.

Examples:

-   cost sensitivity
-   security requirements
-   region preferences

questions\
Blocking clarification questions needed before final design.

confidence\
LLM confidence score representing how well the intent was understood.

------------------------------------------------------------------------

# 2. DesignPlan Schema

DesignPlan represents the **architectural reasoning result** produced by
the internal kre8 design engine.

The engine responsible for generating this plan is:

I2D2 --- Intelligent Infrastructure Design Decision

DesignPlan is **not infrastructure code**.

It represents a **structured architecture design** that can later be
rendered into Terraform or other infrastructure formats.

Example Structure:

    DesignPlan
       intent_summary
       context
       constraints_used
       design_options
       recommended_option
       assumptions
       questions
       resources
       explanation

Field Definitions:

intent_summary\
Normalized summary of the user's request.

context\
Infrastructure context used during reasoning. Examples include existing
clusters, networks, or cloud resources.

constraints_used\
Platform laws applied during reasoning. These follow the **LADE model**:

L --- Limit\
A --- Allow\
D --- Deny\
E --- Exception

design_options\
Multiple viable architecture solutions generated by the engine.

recommended_option\
The design option selected by the reasoning engine.

assumptions\
Assumptions made when full information was not available.

questions\
Clarifications required before rendering or execution.

resources\
High‑level infrastructure components involved in the design.

explanation\
Human‑readable explanation of why the design was chosen.

------------------------------------------------------------------------

# Schema Philosophy

The schema layer enforces a **clean separation between reasoning and
execution**.

Intent → StructuredIntent → DesignPlan → Render → Execute

This ensures:

-   deterministic architecture reasoning
-   traceable design decisions
-   safe infrastructure generation
